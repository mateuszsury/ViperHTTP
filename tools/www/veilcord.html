Why We Built Veilcord

Traditional messaging applications are built around a fundamental flaw: a central server that sees everything. Every message, every file, every call metadata passes through infrastructure controlled by a corporation whose primary business model is the harvesting and monetization of user data. Even services claiming end-to-end encryption often retain metadata ‚Äî who you talk to, when you talk, how often, and from where.

Veilcord was created to fundamentally rethink this model. By combining the Signal Protocol's industry-leading encryption with a true peer-to-peer architecture, we have built a messaging platform where the infrastructure itself cannot be weaponized against users. There are no message databases to subpoena, no metadata logs to analyze, and no corporate entity with the capability to read your conversations.

The signaling server exists only to facilitate initial peer discovery ‚Äî once two users establish a direct connection, all communication flows entirely between their devices. Your encryption keys are generated on your device, stored on your device, and never transmitted to any server. This is zero-knowledge architecture in its purest form.

Signal Protocol: The Gold Standard of Encryption

Veilcord implements the complete Signal Protocol stack ‚Äî the same cryptographic framework that secures billions of messages daily across the world's most trusted communication platforms. But unlike those platforms, Veilcord pairs this encryption with a true peer-to-peer architecture, eliminating the single point of trust that centralized services represent.

The protocol combines Extended Triple Diffie-Hellman (X3DH) key agreement with the Double Ratchet algorithm to provide perfect forward secrecy and future secrecy. This means that even if an attacker somehow obtains your current encryption keys, they cannot decrypt past conversations, and the system automatically heals itself by generating fresh keys for future messages.

  • X3DH key agreement with Ed25519 identity keys and X25519 ephemeral keys
  • Double Ratchet with automatic key rotation on every message exchange
  • AES-256-GCM authenticated encryption for message payloads
  • HMAC-SHA256 for message authentication codes
  • Sender Keys protocol for efficient encrypted group messaging

True Peer-to-Peer: No Server in the Middle

Unlike conventional messaging applications that route every message through corporate infrastructure, Veilcord establishes direct WebRTC connections between communicating peers. Once a connection is established, the signaling server plays no further role in the conversation ‚Äî all data flows directly between the two endpoints.

This architecture provides a critical privacy advantage: there is no central point where messages can be intercepted, logged, or analyzed. Even if the signaling server were compromised, an attacker would only see encrypted connection metadata ‚Äî never the content of any conversation. The WebRTC data channels themselves carry payloads that are already encrypted with the Signal Protocol before transmission, providing defense-in-depth against any network-level adversary.

  • Direct WebRTC connections ‚Äî messages never touch a server after handshake
  • STUN/TURN servers for NAT traversal with ephemeral credentials
  • ICE candidate negotiation via encrypted signaling channel
  • Automatic reconnection with session key persistence
  • Mesh topology for group calls ‚Äî each participant connects directly to every other

Encryption Layers

Four Layers of
Cryptographic Protection

Veilcord implements defense-in-depth with multiple overlapping layers of encryption, each designed to protect against different threat models and attack vectors.

01

Identity Layer: Ed25519 Key Pairs

Every Veilcord user generates a unique Ed25519 identity key pair on first launch. The private key never leaves the device ‚Äî it is stored in the SQLCipher-encrypted database and protected by DPAPI. The corresponding public key serves as your cryptographic identity and is shared with contacts to establish secure channels. Ed25519 was chosen for its excellent security properties, resistance to timing attacks, small key sizes (32 bytes), and extremely fast signature verification. Your Veilcord identity is fundamentally a cryptographic key pair, not an email address or phone number ‚Äî there is no personally identifiable information required to use the platform.

Ed25519 Curve25519 32-byte Keys No PII Required
02

Session Layer: X3DH Key Agreement

When two users initiate a conversation for the first time, the Extended Triple Diffie-Hellman (X3DH) protocol establishes a shared secret through four parallel Diffie-Hellman computations using a combination of long-term identity keys, medium-term signed prekeys, and one-time use ephemeral keys. This protocol provides cryptographic deniability ‚Äî neither party can prove to a third party what the other said ‚Äî while simultaneously providing authentication that the conversation partner is who they claim to be. The resulting shared secret is used to initialize the Double Ratchet algorithm for all subsequent message encryption.

X3DH X25519 DH Signed Prekeys One-time Prekeys HKDF Derivation
03

Message Layer: Double Ratchet Algorithm

Every message is encrypted with a unique key derived from the Double Ratchet algorithm, which combines a symmetric-key ratchet (using HMAC-based key derivation) with a Diffie-Hellman ratchet (generating new key pairs with each message exchange). This provides both perfect forward secrecy ‚Äî compromising a current key cannot decrypt past messages ‚Äî and future secrecy (also called "self-healing") ‚Äî the protocol automatically recovers security after a temporary key compromise. The result is that every single message in a conversation uses a different encryption key, and there is no single "master key" whose compromise would unlock an entire conversation history.

Double Ratchet AES-256-GCM HMAC-SHA256 Per-message Keys Forward Secrecy
04

Storage Layer: SQLCipher + DPAPI

Local data-at-rest protection is provided by SQLCipher, which applies page-level AES-256-CBC encryption to the entire SQLite database. Every page of data is individually encrypted and authenticated with its own HMAC, providing tamper detection in addition to confidentiality. The database encryption key is derived using PBKDF2 with 256,000 iterations and is itself protected by Windows Data Protection API (DPAPI), which encrypts it using credentials tied to the current Windows user session. This two-layer approach means your message history is encrypted at rest without the friction of a master password, while remaining resistant to offline attacks even if the database file is extracted from your device.

SQLCipher AES-256-CBC PBKDF2 256K Iterations DPAPI Per-page HMAC

Privacy Manifesto

Our Uncompromising
Commitment to Privacy

Privacy is not a feature ‚Äî it is the foundation upon which every line of Veilcord's code is built. These are our principles, and they are non-negotiable.

Your Keys, Your Control

Cryptographic identity keys are generated on your device using a cryptographically secure random number generator and stored in an encrypted database that only you can access. At no point during the key generation, storage, or usage process does any key material leave your device or become accessible to any server infrastructure. This is not a policy decision ‚Äî it is an architectural impossibility enforced by the design of the system itself. Even if we wanted to access your keys, we could not. The system was built to make it technically impossible.

No Metadata Collection

Traditional end-to-end encrypted messengers still collect significant metadata: who you communicate with, when and how often you communicate, your IP address, device fingerprint, and location data. This metadata alone can reveal extraordinarily detailed information about your life, relationships, habits, and movements. Veilcord's peer-to-peer architecture eliminates this attack surface entirely. The signaling server sees only ephemeral connection establishment data that is not logged or stored. Once a P2P connection is established, the server has no visibility into the communication whatsoever ‚Äî not even that communication is occurring.

Open Source Transparency

Every line of Veilcord's code is published on GitHub for public review. We believe that security through obscurity is no security at all. By making our source code freely available, we invite the global security community to audit our cryptographic implementations, identify potential vulnerabilities, and verify that our privacy claims are backed by technical reality. The encryption implementation follows published academic specifications exactly, using well-vetted cryptographic libraries rather than custom implementations. We believe you should never have to trust a company's word about your privacy ‚Äî you should be able to verify it yourself.

No Accounts, No Phone Numbers

Veilcord does not require an email address, phone number, or any form of personally identifiable information to create an identity. Your identity on the network is your Ed25519 public key ‚Äî a 32-byte cryptographic value that reveals nothing about who you are in the physical world. Contact discovery happens through direct key exchange between users, not through server-side contact list uploads. This means there is no central directory mapping cryptographic identities to real-world identities, and no mechanism by which an adversary could enumerate or de-anonymize the user base.

Resistant to Compulsion

The architecture of Veilcord is designed such that even under legal compulsion ‚Äî a court order, a national security letter, or a government warrant ‚Äî there is nothing of value to hand over. The signaling server does not log connection metadata. There are no message databases on any server. Encryption keys exist only on user devices. This is not a matter of policy (which can be changed) but of architecture (which cannot be circumvented without fundamentally rebuilding the system). We cannot provide what we do not have, and we have designed the system specifically to ensure that we have nothing to provide.

FAQ

Frequently Asked Questions

Everything you need to know about Veilcord, from technical details to practical usage.

Is Veilcord truly peer-to-peer? What role does the server play? +

Yes, Veilcord is a genuine peer-to-peer application. The signaling server's role is strictly limited to initial peer discovery and WebRTC connection establishment (ICE candidate relay). Once two peers establish a direct connection, all subsequent communication ‚Äî messages, files, voice, and video ‚Äî flows directly between the two devices with zero server involvement. The server never sees message content, and connection metadata is not logged or stored. If the signaling server goes offline after a connection is established, your active conversations continue uninterrupted.

What happens if my device is lost or stolen? +

Your encrypted database is protected by SQLCipher with AES-256 encryption, and the database key is tied to your Windows user session via DPAPI. An attacker who obtains your device but does not have access to your Windows login credentials cannot decrypt your message history. For additional security, you should use full-disk encryption (BitLocker on Windows) and a strong login password. If your device is compromised, generate a new identity on a new device and re-establish contacts ‚Äî the old identity's keys become useless once you stop using them.

How does group encryption work? +

Group messaging uses the Sender Keys protocol. Each group member generates a unique sender key and distributes it to all other group members through individual pairwise Signal Protocol channels. When sending a message to the group, the sender encrypts it once with their sender key, and each recipient can decrypt it using the copy of that sender key they received earlier. This is significantly more efficient than encrypting the message individually for each recipient while maintaining strong security properties. When members join or leave the group, a key rotation occurs to maintain forward secrecy.

Which platforms does Veilcord support? +

Veilcord currently supports Windows (10 and later) as its primary platform. The application is built with Python 3.13+ on the backend and React 19+ on the frontend, with DPAPI integration for secure key storage that is specific to the Windows platform. Cross-platform support (macOS, Linux) is on the roadmap and would require implementing platform-specific secure key storage equivalents (Keychain on macOS, Secret Service on Linux). The React frontend would remain largely unchanged across platforms.

Can I audit the encryption implementation? +

Absolutely. Veilcord is fully open source under a dual license. The complete source code, including all cryptographic implementations, is available on GitHub at github.com/mateuszsury/veilcord. We encourage security researchers and cryptography enthusiasts to review, audit, and report any findings. The encryption implementation follows the published Signal Protocol specifications and uses established cryptographic libraries rather than custom primitives. We believe that verifiable security is the only real security.

Do the AI-powered features (noise cancellation, backgrounds) send data to the cloud? +

No. All AI and machine learning processing in Veilcord runs entirely locally on your device. The DeepFilterNet3 and RNNoise noise cancellation models, MediaPipe body segmentation for virtual backgrounds, and MediaPipe Face Mesh for AR overlays all execute on your local CPU or GPU. No audio or video frames are ever sent to any cloud service for processing. The application automatically detects your hardware capabilities and adjusts processing quality accordingly to maintain smooth performance without relying on external compute resources.
